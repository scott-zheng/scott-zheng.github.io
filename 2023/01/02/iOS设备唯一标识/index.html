1. IDFV

参考链接

供应商标识符 (IDFV) 是开发者为应用指定的代码，设备上属于该开发者的所有应用都拥有同一个 IDFV。

IDFV（Identifier For Vendor）是用于给Vendor标识用户，每个设备在所属同一个Vender的应用里，都有相同的值。形如：95955F33-BFBD-48BA-A630-866D2DAE482D。
根据官方文档，iOS7以后，应用 bundle id 除最后一部分以外相同，则被认为是同一个 Vendor 提供的应用。例如，com.netease.cloudmusic 和 com.netease.playlive 被认为是同一个 Vendor 的应用。

# 使用方法

1) 优点

相较于IDFA，IDFV无需申请权限，可以直接获取

2) 缺点

唯一性有局限
1) 由于IDFV从概念上是属于供应商在设备上的id，所以仅限于在同一vendor下不同应用中进行用户唯一标识。
2) 并且，在用户将同一vendor下所有app删除后，该IDFV将会失效，重新下载该vendor下的app将会生成另一个IDFV

2. 三方库 - SimulateIDFA

参考链接

开源库有两个较常用的 模拟IDFA的库: openIDFA 和 simulateIDFA，都是通过将一系列系统易获取值进行算法加工生成，例如系统容量、系统版本、系统开机时间等。相较之下，simulateIDFA 有更好的表现。

  • SimulateIDFA

一天内这个算法可能的值计算， 24x3600（文件创建时间，单位秒）x 10（文件最后修改时间假设误差在10秒）x 10000000（系统容量误差范围）x 9000000(设备名称范围，这里假设的是每10台就有2个重复，名称范围就是 10000000x0.9)= 77760000000000000000。

设备a的值为 K，那么设备b的值同为K的可能性为： 1/77760000000000000000. 总共有 10000000台设备。因此，这10000000设备中有与a设备的值同为K的可能性为 1/77760000000000000000 x 10000000 = 1/7776000000000。

1) 优点

理论上 SimulateIDFA 的唯一性还是较高的

2) 缺点

其持久性有缺陷。SimulateIDFA分两部分，前16位是在系统升级的时候会发生变化，后16位会在用户的某些行为下可能发生变化（例如：重启手机、修改设备名称、修改手机本地语言）

3. 跨APP共享数据

上述两种方案的思路都是通过直接获取系统提供的唯一标识，或者通过获取系统属性计算唯一标识。但是实际上，还有一种思路就是跨APP共享特定数据来标记用户设备，例如共享用户id、uuid等。例如，应用A网易云音乐在设备中记录自定义ID并上报服务端，在应用B LOOK直播中需要向服务端请求针对设备的定制化广告推荐时，就可以先获取设备中存储的自定义ID并在请求中携带。

1) UIPasteBoard

使用 + (UIPasteboard *)pasteboardWithName:(UIPasteboardName)pasteboardName create:(BOOL)create;

a) 共享范围

同一开发者账号的同一 team ID (APP ID prefix) 下的不同app间可以共享该命名剪切板（待验证）

b) 持久性

官方文档表示应用退出后就会失效，但实际实验应用A退出后，应用B依旧可以获取到剪切内容。该方法持久性有不确定性。

2) KeyChain

Keychain Services 是 macOS 和 iOS 都提供一种安全的存储敏感信息的工具。与此同时，用于认证的证书，密钥，和身份信息，也可以存储在Keychain中，Keychain Services 的安全机制保证了存储这些敏感信息不会被窃取。简单说来，Keychain 就是一个安全容器。

# 特点

a . 数据并不存放在App的Sanbox中，即使删除了App，资料依然保存在keychain中。如果重新安装了app，还可以从keychain获取数据。
b. keychain的数据可以通过group方式，让程序可以在App间共享
c. keychain的数据是经过加密的

虽然 keychain 本意是用于存储安全敏感数据的，但是我们可以利用其设备持久性的特点对之前的方案做一些加工，并利用其数据共享的特点进行跨APP数据共享。

a) IDFV + KeyChain
由上文可知，IDFV的缺点之一在于应用删除后可能不唯一。因此，利用 KeyChain 持久化的特点，在苹果用户第一次安装供应商的任意app时都会在 keyChain 中记录 IDFV，并以此 IDFV 作为后续该设备的唯一标识永久存储，同一Vendor下的其他APP需要时可以加入同一keychain sharing group来获取。
b) 跨APP共享其他数据
keyChain当然也可以做类似 UIPasteBoard 的事情，共享一些自定义的数据

3) App Groups

这个方案可以看作是 UIPasteBoard 的高级方案，持久性更明确，也更贴合语义，符合苹果的期望。此外，相较于 KeyChain，app groups 没有加密。
官方文档说明

AppGroup allows data sharing between two different apps or even app and widgets by creating one common shared path (like document directory). Data saved over there can be accessed by any app which is associated with that particular AppGroup. It is an offline data sharing between apps.

同一 team ID 下的应用可以配置app group共享数据。同一 group 下的应用会共享一个数据空间。需要在应用的 provisioning profile 的 entitlements 配置允许 app groups 以及 group name。

a) 结合 NSUserDefaults，共享少量数据
b) 结合 NSFileManager/NSFilePresenter等等，共享数据文件
